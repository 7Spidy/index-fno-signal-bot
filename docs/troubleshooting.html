Section 01

Morning health check (09:05–09:45 IST)

Run through this in order. Fix the first thing that fails before checking the next.

  • 1
    Morning Login ran green
    Actions tab → Morning Login → latest run has a ✓. Should complete by 09:10 IST. If red, see §2.
  • 2
    Token is in Upstash Redis
    Upstash console → your DB → browse keys → kite:access_token exists and is not expired. If missing, Morning Login failed silently — re-trigger via workflow_dispatch.
  • 3
    Instrument tokens are cached
    Upstash: kite:instrument_tokens key exists. Contains JSON with NIFTY, BANKNIFTY, FINNIFTY, MIDCPNIFTY tokens. If missing, run Morning Login again (it caches instruments in the second step).
  • 4
    Dashboard shows 🟢 Valid (~09:45 IST)
    Open GitHub Pages URL. After the first signal.yml run at ~09:40 IST you should see live prices and indicator values in each instrument card. If still showing dashes (–), see §4.
  • 5
    Signal.yml runs every 5 min
    Actions → Signal Evaluation → runs should be queuing up every 5 minutes between 09:30–15:30 IST. Each run should complete in under 60 seconds.
Section 02

Morning Login failures

⚠
All of these errors were encountered and solved during the initial setup session. The fixes are already in the code. This section documents the progression in case the same errors recur.
🔴
KeyError: 'request_token'
First error encountered. auth.py line ~46.
[auth] ✗ Login failed: 'request_token'

Cause: The original code used allow_redirects=True which made requests try to connect to your app's redirect URL (e.g. https://127.0.0.1) which isn't running in GitHub Actions. The connection failed before request_token could be captured.

Fix appliedReplaced with a custom _InterceptAdapter that intercepts redirects to non-Zerodha URLs before making a TCP connection, extracting request_token from the URL without needing the redirect destination to be reachable.
🔴
"The user is not enabled for the app"
JSON error from /connect/finish endpoint.
{"status":"error","message":"The user is not enabled for the app.","data":null,"error_type":"InputException"}

Cause: The Zerodha Client ID field in the Kite Connect developer app settings was set incorrectly. This field must exactly match the Zerodha user ID used for login.

Fix appliedUpdated the Zerodha Client ID field in kite.trade developer console to match the actual login user ID. The KITE_USER_ID GitHub secret must also match exactly (case-sensitive, no spaces).

If this error returns

1. Go to kite.trade/developers → open your app → check Zerodha Client ID field.
2. Compare to your actual Zerodha login ID (visible at kite.zerodha.com → profile).
3. Both the app field and KITE_USER_ID GitHub secret must be identical.

🔴
request_token not found — stuck on /connect/authorize (POST 405)
React SPA consent page. Required one-time manual browser action.
[auth-debug] Landed on HTML page: https://kite.zerodha.com/connect/authorize?... [auth-debug] POST status=405 location=

Cause: /connect/authorize is Kite's OAuth consent screen — a React SPA. It requires a real browser click (the "Allow Access" button) the very first time an app is authorized by a user. Python's requests library cannot execute JavaScript, so it can't interact with the React app.

This is a one-time setup step only. Once authorized, Kite remembers the consent and subsequent daily logins receive a direct server-side redirect (no consent page) that the code handles automatically.

Fix applied (one-time, already done)Manually opened the authorization URL in a browser and clicked "Allow Access". This only needs to be done once per app+user combination.

If this error returns (it shouldn't)

This would only happen if the app was deleted and recreated (new API key). If so:

1. Open in a browser: https://kite.zerodha.com/connect/login?v=3&api_key=YOUR_API_KEY
2. Log in with Zerodha credentials
3. Click Allow Access on the consent page
4. Browser will redirect to https://127.0.0.1/?request_token=... — this won't load, that's fine, ignore it
5. Trigger Morning Login via workflow_dispatch again

🟡
TOTP / 2FA failure
If KITE_TOTP_SECRET is wrong or expired.
[auth] ✗ Login failed: 403 Client Error or "Invalid TOTP"

Cause: KITE_TOTP_SECRET must be the base32 seed from your authenticator app setup — the string shown when you first scanned the QR code (looks like JBSWY3DPEHPK3PXP). It is NOT the 6-digit rotating OTP.

FixGo to Zerodha profile → Security → 2FA → re-setup TOTP → save the base32 secret shown → update KITE_TOTP_SECRET GitHub secret.

Also check: GitHub Actions server time must be accurate for TOTP to work. It usually is — if TOTP works locally but fails in Actions, the server clock may have drifted (very rare).

Section 03

Signal evaluation failures

⚪
"Outside trading window — exiting"
Normal behaviour — not an error.

This is expected. The script runs from ~09:30 IST but only evaluates signals between 09:40–14:45 IST. Runs before 09:40 and after 14:45 exit immediately with this message. Check also that today is a trading day (not a holiday or weekend).

🔴
"No access token in Redis. Run morning-login.yml."
Morning Login didn't run or failed.

The kite:access_token key is missing or expired (12h TTL). This means Morning Login failed this morning or didn't run.

Fix1. Check Actions → Morning Login for today's run.
2. If it failed, fix the error (see §2) and re-trigger via workflow_dispatch.
3. If it didn't run (missed the cron), trigger manually.
🔴
"ERROR processing NIFTY: ..."
Per-instrument error. Other instruments continue.

The main loop catches per-instrument errors and continues. Common causes:

  • Kite API rate limit — too many requests (unlikely with 4 instruments)
  • Instrument token expired (futures rolled over to next month) — Morning Login resolves this daily
  • Market not open yet — candles return empty before 09:15 IST
  • Kite API downtime — check status.zerodha.com
🟡
Signals firing too often / not at all
Indicator calibration. Not an error.

The dashboard shows which conditions (C1–C4) pass and fail for each instrument on every tick. If all 4 are passing very frequently, consider tightening the config:

DI_THRESHOLD = 25          # increase to 28-30 for stricter DMI
RSI_SLOPE_LOOKBACK = 3     # increase to 4-5 for stricter RSI
VWAP_CROSS_WINDOW_CANDLES = 6  # decrease to 3-4 for tighter cross window
COOLDOWN_CANDLES = 3       # increase to prevent rapid re-firing

Edit src/config.py, commit, push. Changes take effect on the next run.

Section 04

Dashboard not updating

🟡
Dashboard shows stale data or dashes (–)

Check 1: Has signal.yml run today?

Actions → Signal Evaluation → look for runs today during market hours. If no runs, the cron may not have triggered yet or there were failures.

Check 2: Is GitHub Pages enabled?

Repo Settings → Pages → Source must be set to main branch, /docs folder. The URL will be https://7spidy.github.io/index-fno-signal-bot.

Check 3: Browser cache

GitHub Pages CDN caches aggressively. The dashboard already uses ?t=Date.now() to bust cache on the JSON fetch, but try a hard refresh (Ctrl+Shift+R) on the dashboard page itself.

Check 4: dashboard.json was committed

Repo → code tab → docs/dashboard.json → check the "last committed" timestamp. If it's old, the git push from dashboard_writer.py may have failed. Check the signal.yml run logs for git errors.

🔴
"Dashboard offline — retrying" banner

The browser couldn't fetch dashboard.json. Possible causes:

  • GitHub Pages not enabled (see Check 2 above)
  • You're opening docs/index.html as a local file (file://) — it must be served from GitHub Pages or a local HTTP server
  • GitHub Pages deployment still in progress (can take 1-2 minutes after a push)
Section 05

Discord alerts not arriving

⚪
No Discord messages all day
This is normal if no signals fired.

Discord only fires when ALL 4 conditions pass simultaneously for CE or PE. Check the dashboard history log — if all dots are mostly dim (grey), conditions just haven't aligned. This is expected behaviour, not an error.

ℹ
The dashboard updates silently every 5 minutes whether or not a signal fires. Silence on Discord = no signal. Check the dashboard to confirm the bot is running.
🔴
Signal showed on dashboard but no Discord message

Check signal.yml run logs for [notifier] lines. Possible causes:

  • DISCORD_WEBHOOK_URL secret is missing or incorrect — check GitHub Secrets
  • The webhook was deleted in Discord — recreate it: channel → Edit → Integrations → Webhooks → New Webhook
  • Discord API was temporarily down
  • The signal was dedup-suppressed (same candle timestamp fired twice within the 5-min window)
Section 06

Upstash Redis issues

🔴
Redis operations failing silently

src/state.py never raises — all errors are printed and return None/False. Check signal.yml logs for lines like [state] redis_get(...) error:.

Verify your Upstash connection manually

Open this URL in your browser (replace values):

https://YOUR_UPSTASH_URL/get/kite:access_token
Authorization: Bearer YOUR_TOKEN

Or use curl:

curl https://YOUR_UPSTASH_URL/get/kite:access_token \
  -H "Authorization: Bearer YOUR_TOKEN"

If you get a JSON response, the connection is fine. If you get 401, the token is wrong.

Section 07

Kite Connect configuration

🟡
Monthly futures rollover (last Thursday of each month)

Nifty futures expire on the last Thursday of each month. Morning Login re-resolves instrument tokens daily (resolve_futures_tokens() always picks the nearest-expiry contract), so this is handled automatically. No action needed.

🟡
Kite Connect subscription renewal (every 3 months)

The app expires on 03 Jul 2026 (visible in kite.trade developer console). Renew before it expires — the morning login will start failing with "subscription expired" type errors if it lapses.

⚠
Set a calendar reminder for 27 Jun 2026 to renew the Kite Connect subscription at kite.trade before it expires on 03 Jul 2026.
🟡
NSE holidays not in holidays_2026.json

The bot uses holidays_2026.json to skip NSE holidays. Some dates are marked as tentative (Bakri Id, Dussehra, Diwali depend on lunar calendar). Verify the list against the official NSE holiday circular before those dates at nseindia.com.

Cron schedule reference

⚠
GitHub Actions cron jobs can be delayed by up to 15 minutes during high load periods. Morning Login running at 09:08 instead of 09:05 is normal. Signal runs at :07 instead of :05 is normal.

GitHub Actions cron note

GitHub disables scheduled workflows on repos with no activity for 60 days. If the bot stops running after a period of inactivity, go to Actions → Morning Login → click Enable workflow. Alternatively, re-trigger manually via workflow_dispatch to wake it up.
